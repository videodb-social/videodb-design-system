# Project memory — VideoDB Design System

Notes for any Claude session working in this folder. Read before touching code.

## What this project is

A self-contained design system for **videodb.io**:

- **`videodb-design.md`** — the canonical spec. Versioned in the frontmatter (currently `videodb-design-v1.5`). Bump the version stamp + `status:` line when you change anything substantive.
- **`homepage-showcase.html`** — a single-page interactive showcase of every component in the spec. The deployed prod copy is built from this via `scripts/build-vercel.py`.
- **`assets/logos/`** — official VideoDB logo PNGs. Referenced by the showcase and any consuming site. **Never re-render the wordmark in HTML/CSS — always use the PNG.**
- **`scripts/build-vercel.py`** — strips the review-mode harness from the showcase, copies assets, and writes a `vercel.json`, producing a deploy-ready `vercel-deploy/` folder.
- **`scripts/ui-review-flow/`** — reusable Figma-style review overlay (see "Review flow" section below).
- **`feedback/`** — transient review-state. Gitignored. Holds `feedback-bundle-LATEST.json` (user → AI) and `response-bundle-LATEST.json` (AI → user), plus extracted attachments per comment ID.
- **`vercel-deploy/`** — derived build output. Gitignored. Regenerated by `scripts/build-vercel.py`.

Deployed at: https://videodb-design-system.vercel.app/design-system  
Source repo: https://github.com/videodb-social/videodb-design-system

## Hard rules (from the spec — non-negotiable)

When making any change that touches the showcase or design.md:

- **Geist + JetBrains Mono only.** Geist for display/body/UI; JetBrains Mono for chrome (codes, badges, code blocks).
- **Brand orange `#F24E1E` is the single accent.** Hover `#D14016`. No second accent color.
- **Semantic error red `#E5484D`** — for invalid input / destructive states only. Never for general accents.
- **Sections fully commit to dark or light.** No transitional grey states. Dark = `#0A0A0A` (hero/divider) and `#161616` (cards-in-light, code blocks, mega-menus). Light = `#F5F5F7` (section bg) and `#E8E8EA` (soft cards).
- **Two-tone declarative headings** with `<br>` and `<span>` — never stacked `<div>` blocks. Exception: primary hero is singular white (not two-tone).
- **No emoji anywhere.** Solar icons (Iconify) for everything.
- **display-md `line-height: 1.15 !important`** — current floor after multiple iterations. Anything tighter risks descender collisions.
- **Per-block code animation state via WeakMap.** Multiple code blocks on one page use class selectors (`.typed-out`, `.cursor`), not IDs, so they coexist.
- **In running prose write `VideoDB`** (capital V, capital DB). The URL `videodb.io` keeps lowercase.

For everything else, **read `videodb-design.md` first** — it's the source of truth.

## Review flow — how design iteration works on this project

This project uses a Figma-style point-and-click review overlay built into the showcase. The harness is gated by `?review=1` and lives in `scripts/ui-review-flow/` as a reusable bundle.

### Workflow each round

1. User opens `homepage-showcase.html?review=1` locally.
2. User clicks elements to drop pins, leaves comments + attachments, hits **Submit feedback**.
3. The JSON bundle (`feedback-bundle-LATEST.json`) lands in `feedback/`.
4. User tells Claude "the bundle is ready".
5. Claude reads the bundle, extracts attachments to `feedback/attachments/<comment-id>/`, processes each open/needs-review comment, makes the requested changes to `homepage-showcase.html` and/or `videodb-design.md`.
6. Claude writes `feedback/response-bundle-LATEST.json` with `{ comment_id, new_status, reply }` per addressed comment.
7. User clicks **Load Claude responses…** in the sidebar — pin statuses flip to `needs-review` (blue) with replies attached.
8. User marks each comment resolved or pushes back with new comments; resubmits the bundle.
9. Loop until everything's resolved.

### Files involved

- **Source:** review harness is inside `homepage-showcase.html` (CSS in the `<style>` block, JS in its own `<script>` block before `</body>`).
- **Reusable bundle:** `scripts/ui-review-flow/` contains a standalone copy:
  - `SKILL.md` — short skill description (when/why to use the flow).
  - `INTEGRATION.md` — long-form how-to and bundle schema reference.
  - `review-mode.css` — drop-in CSS.
  - `review-mode.js` — drop-in JS.
  - `extract.py` — re-extracts the above from `homepage-showcase.html` if the code evolves.

### Reusing the review flow on other projects

The `scripts/ui-review-flow/` bundle is self-contained. To make it available across all Claude sessions on other projects:

```bash
cp -r scripts/ui-review-flow ~/Documents/Claude/Skills/
```

Then in any future session say:

> "Use the UI review flow skill at `~/Documents/Claude/Skills/ui-review-flow/`."

And Claude will load `SKILL.md`, find the CSS/JS, and inject them into whatever showcase you're working on.

### Updating the bundle after spec changes

If you modify the review harness inside `homepage-showcase.html`, re-export the standalone bundle:

```bash
python3 scripts/ui-review-flow/extract.py
```

## Deployment

```bash
# Refresh prod build
python3 scripts/build-vercel.py

# Push to GitHub (auto-deploy will need to be wired up separately)
git add . && git commit -m "..." && git push

# Manual Vercel push from the deploy folder
cd vercel-deploy && vercel --prod
```

The `vercel.json` in the deploy folder sets `cleanUrls: true` and redirects `/` → `/design-system`. The deployed showcase is the **stripped production version** (review harness removed). Review flow only works on the local source HTML, never on the deployed site.

## Pre-publish checklist

Before pointing real users at the production deploy:

- Bump the `version` and `status` fields in `videodb-design.md`.
- Run `python3 scripts/build-vercel.py` to refresh `vercel-deploy/`.
- Clear any open comments in `feedback/` (or just delete the folder — it's gitignored).
- `cd vercel-deploy && vercel --prod`.

## Versioning

The spec is versioned. Major component changes → bump the minor (`v1.5` → `v1.6`). Token / typography / motion baseline changes → bump the major. Always edit the `status:` line in the frontmatter to summarize what changed; that's the human-readable changelog.

Ancestor / superseded design docs are stored in sibling folders, prefixed with `V0.x` so they're not accidentally referenced:

- `V0.1-neu-videodb-DESIGN.md` (Nexus/neuform ancestor)
- `V0.1b-Platform-Architecture-Hero-DESIGN.md` (Platform Architecture ancestor)
- `V0.2-NEO-videodb-design.md` (NEO v1.1 — immediate predecessor to current v1.5)
- `V0.2.1-NEO-videodb-design.md` (duplicate copy of V0.2)

Don't reference any of these for design decisions. They exist for archaeology only.
